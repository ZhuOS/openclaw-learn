# Ch05 · 记忆体系

> 一个只有上下文窗口的模型没有"昨天"。OpenClaw 给 Agent 装上了**四层记忆**——这是把 LLM 从"有状态对话"升级为"有身份感的长期搭档"的关键工程。

---

## 5.1 为什么需要一整章来讲"记忆"

在 Ch04 里你看到了 Pi Agent 的核心循环。但有一个关键事实**必须单独强调**：

> **模型本身没有记忆。每次 API 调用都是无状态的。**

你每次向 Claude / GPT / Gemini 发请求，它读的都只是你这次塞进 `messages` 数组里的内容。它**不知道**：
- 你们上周聊过什么
- 你的名字叫 Mason
- 你偏好直接表达、不爱废话
- 它自己叫什么、性格是什么

让 Agent 具备这些"记忆"，**完全是 Agent 框架（而不是模型）的责任**。OpenClaw 在这件事上做得**非常彻底**——它把记忆拆成了四个层次，每层解决一类问题。

```
┌──────────────────────────────────────────────────────────┐
│  四层记忆全景                                              │
│                                                           │
│  ① 瞬时记忆   =  当前请求的上下文窗口                      │
│     ↕                                                     │
│  ② 会话记忆   =  JSONL 消息树（线性 + 分支 + 压缩）         │
│     ↕                                                     │
│  ③ 人设记忆   =  AGENTS.md / SOUL.md / USER.md             │
│     ↕                                                     │
│  ④ 长期记忆   =  QMD 向量检索（跨会话、跨 Agent）            │
│                                                           │
│  此外还有：Workspace 作为"物理记忆"（文件系统）              │
└──────────────────────────────────────────────────────────┘
```

这四层**不是互相替代**，而是**职责不同**：

| 层 | 生命周期 | 载体 | 谁来维护 |
|---|---|---|---|
| ① 瞬时 | 一次 LLM 调用 | `messages[]` 内存数组 | Pi SDK 自动管理 |
| ② 会话 | 单个对话上下文 | JSONL 文件 | `SessionManager` |
| ③ 人设 | Agent 生命周期内稳定 | Markdown 文件 | 用户/系统主动维护 |
| ④ 长期 | 永久 | QMD 向量库 | 自动索引 + 手动增删 |

接下来一层一层拆。

---

## 5.2 瞬时记忆：上下文窗口的真相

这是最基础的一层，也是最容易被误解的一层。

### 5.2.1 LLM 看到的其实只有这些

当 Pi Agent 要调一次 LLM，它组装的 request body 简化来看是：

```json
{
  "model": "claude-sonnet-4-5-20250929",
  "system": "<拼接后的 system prompt>",
  "messages": [
    { "role": "user",      "content": "..." },
    { "role": "assistant", "content": "..." },
    { "role": "user",      "content": "..." },
    ...
  ],
  "tools": [ ... ]
}
```

**关键点**：每次调用都要**把完整的 `system` 和 `messages` 重新传一遍**。所谓"模型记得我们聊过什么"，本质是 Agent 框架**每次把历史塞回去**。

### 5.2.2 OpenClaw 做的实际工作

Pi Agent 的核心是一个 tool-calling 循环（见 Ch04.7）。这个循环在运行时：

1. **首次 prompt**：从会话 JSONL 加载历史 → 渲染 system prompt（含人设记忆） → 组装 messages → 调 LLM
2. **工具回流**：tool 执行结果作为 `role: "tool"` 消息追加到 messages → 再次调 LLM
3. **超限处理**：捕获 `context length exceeded` → 触发压缩（见 5.3.4）→ 重新 prompt

### 5.2.3 为什么这一层被"藏起来"了

Pi SDK 把这一层**完全封装**，你在 OpenClaw 代码里看不到显式的 `messages.push()` 操作。这是**好事**——意味着你维护 Agent 时不用操心"这条消息要不要进上下文、放哪个位置"。

**但你要知道它的存在**，因为后面三层的设计动机都是"**让瞬时层尽可能高效地装下有用信息**"。

## 5.3 会话记忆：JSONL 树

### 5.3.1 物理存储

每个 Agent 的会话都在：

```
~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl
```

JSONL = **JSON Lines**，每行一个 JSON 对象，追加写友好：

```jsonl
{"id":"msg-001","parentId":null,      "role":"user","content":"hi","ts":1714234567000}
{"id":"msg-002","parentId":"msg-001", "role":"assistant","content":"hello","ts":1714234568000}
{"id":"msg-003","parentId":"msg-002", "role":"user","content":"搜一下 OpenClaw","ts":1714234580000}
{"id":"msg-004","parentId":"msg-003", "role":"assistant","content":"...(tool call)...","ts":1714234581000}
```

### 5.3.2 为什么是"树"不是"数组"

看到 `id` / `parentId` 字段了吗？这是让会话**支持分支**的关键设计：

```
msg-001 (user: hi)
  └─ msg-002 (assistant: hello)
       └─ msg-003 (user: 搜一下 OpenClaw)
            ├─ msg-004 (assistant: 找到了...)      ← 主线
            └─ msg-004-alt (assistant: 换个搜法...) ← 重新生成的分支
                 └─ msg-005-alt (user: 这个好)
```

**实际用途**：

| 场景 | 怎么用树 |
|---|---|
| 重新生成回答 | 新建一条 `assistant` 消息，`parentId` 指向同一个 user 消息 |
| 回溯到某个节点重开 | 从某条 user 消息拉出新分支 |
| Checkpoint | 标记某个节点为"保存点"，之后可以回滚 |
| A/B 对比 | 同一 prompt 产生两个回复，并排比较 |

**对比**：绝大多数 chatbot（包括 ChatGPT 早期）用的是**线性数组**，"重新生成"就会覆盖前一个。OpenClaw 的树结构让**所有历史都是不可丢的**，这对长期协作至关重要。

### 5.3.3 `SessionManager` 的缓存策略

每次都全量解析 JSONL 显然不现实，所以 `packages/sessions/src/manager.ts`（典型位置）会做：

```ts
// 伪代码
class SessionManager {
  private cache = new Map<string, SessionState>();

  async load(sessionId: string): Promise<SessionState> {
    if (this.cache.has(sessionId)) return this.cache.get(sessionId)!;
    const state = await parseJsonl(this.pathOf(sessionId));
    this.cache.set(sessionId, state);
    return state;
  }

  async append(sessionId: string, msg: Message): Promise<void> {
    const state = await this.load(sessionId);
    state.messages.push(msg);
    state.byId.set(msg.id, msg);
    // 追加写 JSONL（不是重写）
    await fs.appendFile(this.pathOf(sessionId), JSON.stringify(msg) + "\n");
  }
}
```

**关键决策**：

- **追加写**：只 append，不改写已有行。这让 JSONL 文件天然支持并发读 + 单一写
- **按需加载**：LRU 或简单的"用到才加载"策略，避免启动时扫全部历史
- **内存映射**：大会话可能用 mmap，但常规 < 50MB 的 JSONL 直接加载就行

### 5.3.4 自动压缩（compaction）

会话越聊越长，迟早撞 context window 上限（Claude 200K、Gemini 1M、但即便 1M 也有被超过的一天）。OpenClaw 的策略：

```
│    对话累积
│      ↓
│    LLM 返回 "context length exceeded" / "request too large"
│      ↓
│    错误分类器 isContextOverflowError() 命中
│      ↓
│    emit("compaction_start")
│      ↓
│    调用 LLM 对历史做结构化摘要
│      ↓
│    把旧消息替换为一条 "[summary]..." 消息
│      ↓
│    emit("compaction_end")
│      ↓
│    重新 prompt 原问题
```

**代码定位**：`packages/agent/src/compaction/` 附近。关键函数：

- `compactEmbeddedPiSessionDirect(session, { tokenBudget })` — 直接压缩当前会话
- `computeAdaptiveBudget()` — 根据模型 context window 算出目标预算

### 5.3.5 两个关键 Hook

这是 OpenClaw **在 Pi SDK 之外加的私货**，在 `packages/agent/src/hooks/`：

**① `compaction-safeguard`**

```
触发：压缩完成
检查：压缩后 token 是否仍然 > 安全线（比如 80% context window）
行动：如果仍超 → 再压一次，目标预算减半
极端：三次仍超 → 抛 CompactionFailureError，让用户介入
```

这个护栏的意义：**防止压缩不彻底导致下一轮 prompt 立刻又爆**。没有它，系统会陷入"压缩→爆→压缩→爆"的死循环。

**② `context-pruning`**

```
触发：每次 turn 开始前
检查：消息上是否有 `cache_ttl` 元数据
行动：TTL 过期的消息标记为"可丢弃"，压缩时优先处理
```

这个 hook 让"临时信息"（比如查了一次天气的工具返回值）**不会长期占用上下文**。

### 5.3.6 压缩后信息找得回吗？

**找得回——因为 JSONL 是追加写的**。

压缩只改变**当前 in-memory 的 messages 数组**（瞬时层），**不会删除 JSONL 文件里的原始消息**。如果你需要回溯，完全可以：

```bash
# 查看某个会话的完整原始历史
cat ~/.openclaw/agents/<agentId>/sessions/<sessionId>.jsonl | jq .
```

**⚠️ 例外**：如果用户主动执行 `sessions_prune` 工具（如果 Agent 有这个权限），原始记录可能被清理。默认权限下不开放。

### 5.3.7 手动压缩

作为用户你也可以在 Control UI 或通过工具调用主动触发：

```bash
# 假设启用了 sessions_compact 工具
"帮我压缩当前会话"
```

Agent 会执行 `sessions_compact({ target_tokens: 50000 })`，返回压缩前后的 token 数和摘要长度。

## 5.4 人设记忆：三个 Markdown 文件的分工

这一层是**OpenClaw 独特的设计**——它把"Agent 是谁、服务谁、做什么"**外化成文件**，而不是写死在代码里。

### 5.4.1 三个文件在哪里

每个 Agent 的 workspace 根目录：

```
~/.openclaw/workspace-<agentId>/
├── AGENTS.md     # Agent 的"职责说明书"
├── SOUL.md       # Agent 的"性格与表达风格"
├── USER.md       # Agent 眼中的"用户画像"
├── TOOLS.md      # 自定义工具的使用指南（可选）
└── ...（工作区其他文件）
```

**装载机制**：`DefaultResourceLoader`（Pi SDK 提供）在会话创建时自动读取这几个文件，拼进 system prompt 的不同区块。

### 5.4.2 三个文件各自的职责

| 文件 | 定位 | 典型内容 |
|---|---|---|
| **AGENTS.md** | 做什么 / 边界在哪 | "我是家庭事务助理；不处理工作邮件；不主动调 exec" |
| **SOUL.md** | 怎么说话 / 什么性格 | "直接、有态度、不用套话；用🦁作为签名 emoji" |
| **USER.md** | 服务的对象是谁 | "Mason，深圳程序员，二胎爸爸，做 AI 内容创作" |

### 5.4.3 实际例子（取自 Leo 🦁）

**`SOUL.md`**（你的 AI 搭档的灵魂）：

```markdown
---
title: "SOUL.md"
read_when:
  - Every session start
---

# SOUL.md - Who I Am

我是 Leo 🦁。Mason 的 AI 搭档。

## Core Truths
- 直接有用，不演
- 有态度
- 先自己搞定，搞不定再问
- 推 Mason 去行动
- 务实不鸡汤
```

**`USER.md`**（关于服务对象）：

```markdown
---
summary: "User profile record"
---

# USER.md - About Your Human
- Name: Mason Zhu
- City: 深圳
- Notes: 程序员，专注 AI Agent 领域。务实、直接，不喜欢虚的。二胎爸爸。
```

**`AGENTS.md`**（职责边界）：

```markdown
# AGENTS.md

## 做什么
- 技术问题拆方案，带答案来
- 内容创作（公众号 / 小红书）辅助

## 不做什么
- 私人的东西不外泄
- 对外操作（发消息、发邮件）先确认
- 不代替 Mason 在群聊里说话
```

### 5.4.4 注入顺序与优先级

system prompt 的最终形态大致是：

```
┌─────────────────────────────────────┐
│ [Pi 内置基础指令]                    │
│   工具调用规范、输出格式要求等        │
├─────────────────────────────────────┤
│ [AGENTS.md 内容]                    │ ← 职责/边界
├─────────────────────────────────────┤
│ [SOUL.md 内容]                      │ ← 人设/风格
├─────────────────────────────────────┤
│ [USER.md 内容]                      │ ← 用户画像
├─────────────────────────────────────┤
│ [激活的 SKILL.md 内容]               │ ← 见 Ch07
├─────────────────────────────────────┤
│ [TOOLS.md 内容（如果有）]            │ ← 工具使用指南
├─────────────────────────────────────┤
│ [OpenClaw 渠道上下文覆盖]            │ ← applySystemPromptOverrideToSession
│   "你现在在 WhatsApp 群聊 xxx 里..." │
└─────────────────────────────────────┘
```

**优先级规则**：**后写的覆盖先写的**。这意味着：
- 渠道动态覆盖 > TOOLS.md > SKILL.md > USER.md > SOUL.md > AGENTS.md > Pi 基础指令

**⚠️ 设计哲学**：越"动态 / 场景相关"的内容写在越后面，让它能**压过**通用人设。

### 5.4.5 `applySystemPromptOverrideToSession` 的作用

这是 OpenClaw **在 Pi 之上加的扩展点**（Ch04.4 提过）。它干的事：

```ts
// 伪代码
function applySystemPromptOverrideToSession(session, opts) {
  const base = session.systemPrompt;
  const channelCtx = renderChannelContext(opts.channel, opts.peer);
  // 例如：channelCtx = "你当前在 Telegram 群聊 '家人群' 里，
  //                    发送者是 Mom，她用中文聊天。"
  session.systemPrompt = base + "\n\n" + channelCtx;
}
```

**为什么不能把这部分写进 SOUL.md**：因为它**每条消息都可能不同**（不同群、不同对话方）。把静态的（SOUL/USER/AGENTS）和动态的（渠道上下文）分开，才能让人设记忆**跨渠道复用**。

## 5.5 长期记忆：QMD 向量检索

### 5.5.1 QMD 是什么

**QMD** = Qdrant-backed Markdown Database（OpenClaw 自己的命名）。底层是 [Qdrant](https://qdrant.tech/)（开源向量数据库），外壳做了面向 Markdown 会话记录的封装。

功能：**把 JSONL 会话 / Markdown 文件 / 任意文本 按语义切片、向量化，支持按含义检索**。

### 5.5.2 为什么需要这一层

会话记忆（JSONL 树）解决**单个对话内的连续性**。但下面这些场景它搞不定：

- "上个月我们是怎么决定这个方案的？" → 跨会话
- "Mason 以前提过他爸爸喜欢吃什么来着？" → 跨话题检索
- "我在 family agent 里聊过的家庭预算，work agent 能看到吗？" → 跨 Agent

QMD 就是为这些场景设计的。

### 5.5.3 工作流程

```
① 索引阶段（后台异步）
   会话 JSONL 追加新消息
      ↓
   embedding 模型生成向量
      ↓
   写入 Qdrant collection
        collection 命名：agent-<agentId>-sessions

② 检索阶段（Agent 主动）
   Agent 调 memory_search({ query: "爸爸喜欢的菜" })
      ↓
   Query 向量化 → Qdrant 相似度检索
      ↓
   返回 top-K 消息片段（带 sessionId + msgId）
      ↓
   Agent 把结果作为上下文再调 LLM
```

### 5.5.4 配置示例

```json
{
  "agents": {
    "list": [{
      "id": "main",
      "memorySearch": {
        "qmd": {
          "extraCollections": [
            { "path": "~/agents/family/sessions", "name": "family-sessions" }
          ]
        }
      }
    }]
  }
}
```

- `main` Agent 默认索引自己的 sessions
- `extraCollections` 开一个**只读**窗口，让 `main` 能搜到 `family` 的历史

### 5.5.5 触发方式

Agent 调用由插件/内置提供的工具：

| 工具 | 作用 |
|---|---|
| `memory_search({ query, limit })` | 按语义搜记忆 |
| `memory_get({ messageId })` | 按 id 取具体消息 |
| `memory_summarize({ topic, since })` | 对某主题做汇总（调 LLM） |

这些工具在 **`group:memory`** 组里（见 Ch07.2）。默认不启用，要在 `tools.allow` 里显式打开。

### 5.5.6 存储位置

```
~/.openclaw/qmd/
├── qdrant/              # Qdrant 本地数据目录
└── index-state.json     # 哪些 sessions 被索引过、上次索引到哪一行
```

Qdrant 以**本地 embedded 模式**运行（默认），不需要单独起服务。如果你的 Agent 非常活跃，几十万条消息后 QMD 占盘可能到 GB 级别，值得监控。

### 5.5.7 ⚠️ 四个坑

1. **embedding 模型需要额外 API Key**。默认走 OpenAI `text-embedding-3-small`，你要在 providers 里配。
2. **索引是异步的**。刚聊完的消息，可能要几秒到几十秒才能搜到。
3. **跨 Agent 共享只读，没有写权限**。`main` 读 `family-sessions`，但不能往里塞东西。
4. **向量检索不是"精确搜索"**。"爸爸喜欢吃什么"可能把"爸爸不喜欢吃的"也搜出来，因为语义接近。

## 5.6 Workspace：作为"物理记忆"

这是第五层（不在主图四层里，但同样重要）：**文件系统本身就是记忆**。

### 5.6.1 每个 Agent 的工作区

```
~/.openclaw/workspace-<agentId>/
├── AGENTS.md / SOUL.md / USER.md       # 人设记忆（见 5.4）
├── notes/                              # Agent 做笔记的地方
│   └── 2026-04-28.md
├── projects/                           # 项目文件
├── cache/                              # 中间结果
└── downloads/                          # 下载产物
```

这个目录**默认就是 Agent 的 cwd**。Agent 调 `read` / `write` / `exec` 时，相对路径都从这里开始。

### 5.6.2 作为记忆的三种模式

**① Markdown 笔记模式**

让 Agent 自己往 `notes/YYYY-MM-DD.md` 里记东西。下次启动时，**工作区里的 Markdown 不会自动进 system prompt**，但 Agent 可以 `read` 它们。

事实上你的 WorkBuddy 就是这样做的——每次开新 session，我都会读 `memory/YYYY-MM-DD.md` 和 `memory/MEMORY.md` 来获取昨天的上下文。这和 OpenClaw Agent 的机制**同构**。

**② 生成物持久化**

Agent 帮你写了报告、生成了图片、拉了数据——这些文件留在 workspace 里，下次会话可以继续用。

**③ 缓存模式**

Tool 的结果可以落到 `cache/`（比如 Ch09 的天气插件的 10 分钟缓存，也可以持久化到文件而不是内存 Map）。

### 5.6.3 与沙箱的关系

**⚠️ 重要**：如果 Agent 用 Docker 沙箱运行（`sandbox.mode: "all"`），它看到的"工作区"是**挂载到容器里**的同一个目录。所以：

- 容器里的写入 = 宿主机的写入（有持久化）
- 容器里的 `/tmp` = 不持久化（容器重启就丢）

建议：**明确让 Agent 把要保留的东西写到工作区根目录下的子目录**，而不是 `/tmp`。

## 5.7 四层协作实例

用一个具体场景把四层串起来。假设 Mason 对 Leo 说：

> "上次我们聊过哪个 VAD 模型性能最好？"

**Leo 处理这条消息时，四层记忆是这样协作的**：

```
① 瞬时记忆：Pi Agent 收到 prompt，从当前 session 拉出最近 N 条消息放 messages[]

② 会话记忆：SessionManager 检查当前 session 的 JSONL
   - 最近 10 条里没提过 VAD → 得去更远的地方找

③ 人设记忆：system prompt 里已经注入 USER.md
   - 模型知道 Mason 是语音 VAD 工程师
   - 知道他会问技术细节，不用做基础科普

④ 长期记忆：模型决定调工具
   → memory_search({ query: "VAD 模型性能对比", limit: 5 })
   → 返回 3 个月前某次聊天里的具体对比数据
   → 作为 tool result 回流到 messages[]
   → 模型生成最终答案："Silero-VAD 在 NER_470_long 数据集上..."

⑤ Workspace（物理记忆）：如果那次对话有生成过 ab_test_report.md
   模型还可以 read 它来确认细节
```

**缺任何一层都不行**：
- 缺 ①：模型没法生成回答
- 缺 ②：连"上次"这个词指的什么会话都不知道
- 缺 ③：模型不知道 VAD 是用户的专业领域
- 缺 ④：3 个月前的内容早被压缩掉了，搜不到
- 缺 ⑤：关键数据如果只在文件里，不读就丢

## 5.8 踩坑与最佳实践

### 5.8.1 记忆膨胀

**问题**：聊久了会话 JSONL 几十 MB，QMD 向量库上 GB，加载慢、检索慢、成本高。

**应对**：
- **启用 context-pruning hook**：让工具结果这类"临时信息"自动过期
- **定期 checkpoint + fork**：重要节点单独存一份，旧会话归档
- **分 session**：不要一个 Agent 只用一个 session，按话题/项目拆

### 5.8.2 人设漂移

**问题**：SOUL.md 写了"不说客套话"，但模型聊着聊着又开始"好的！没问题！"

**原因**：往往是你在 prompt 里不小心塞了客套示例，或者激活的 Skill 里有反例。

**应对**：
- 周期性审查 SOUL.md + 所有 SKILL.md 的一致性
- 开 `OPENCLAW_DEBUG_PROMPT=1` 打印实际 system prompt，对照检查

### 5.8.3 跨 Agent 记忆泄漏

**问题**：开了 `extraCollections` 让 main 读 family，结果 main 把家人隐私讲出去了。

**应对**：
- **权限最小化**：只开必要的 collection
- **在 AGENTS.md 写死边界**："即使能搜到，也不要把家人隐私在工作渠道讨论"
- 用 Ch08 的**渠道路由**兜底：敏感信息的群，根本不给 main 回复机会

### 5.8.4 压缩后找不到关键信息

**问题**：压缩把关键对话摘要成了一句话，模型后来答非所问。

**应对**：
- 重要决策用 `sessions_pin`（如果有此工具）固定不压
- 或者让 Agent 主动把结论写入 workspace 的 Markdown，**物化**后就不怕压缩了

### 5.8.5 ⚠️ 生产级最佳实践清单

```
✅ SOUL.md / USER.md / AGENTS.md 三个文件都有
✅ 三个文件都在 git 里，变更可追溯
✅ compaction-safeguard hook 启用
✅ context-pruning hook 启用
✅ QMD collection 按 Agent 隔离，跨访问走 extraCollections 只读
✅ 重要 session 定期备份（就是 cp 那个 JSONL 文件）
✅ workspace 定期快照（rsync / snapshot）
```

## 5.9 设计哲学小结

从这一章能提炼出 OpenClaw 记忆体系的**五条核心原则**：

1. **分层明确** — 每层职责单一，不互相侵占
2. **持久化优先** — 能落盘就落盘（JSONL 追加写、Markdown 文件、Qdrant 索引）
3. **树而非数组** — 会话用树结构，不丢任何历史分支
4. **外化人设** — 性格和边界写在文件里，不写在代码里
5. **显式共享** — 跨 Agent 访问必须配置，默认隔离

这些原则在**所有工业级 Agent 框架**里都适用——你自己写的任何长期运行的 AI Agent，都可以照这一章的分层去设计。

---

## 5.10 本章小结

- OpenClaw 的记忆分**四层**：瞬时 / 会话 / 人设 / 长期，外加 Workspace 作为物理记忆
- **会话层**用 JSONL 树存储，追加写、支持分支、按需加载
- **压缩机制**由 `compaction-safeguard` + `context-pruning` 两个 hook 护栏
- **人设层**外化为 `AGENTS.md` / `SOUL.md` / `USER.md` 三个 Markdown 文件，注入 system prompt
- **长期层**用 QMD（Qdrant + Markdown 封装）做语义检索，跨 Agent 只读共享
- 五条设计原则：分层明确 / 持久化优先 / 树而非数组 / 外化人设 / 显式共享
