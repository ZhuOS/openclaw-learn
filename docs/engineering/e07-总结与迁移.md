# E07 · 把这套方法论搬到你的项目

> 总结一下：从 OpenClaw 学到的，怎么用到你自己的 Agent 上。

## 7.1 6 条核心方法论

读完前 6 章，你应该已经形成了这样的思考框架：

| 维度 | 模型本身的局限 | 工程化解法 | 关键洞察 |
|---|---|---|---|
| **记忆** | 完全无状态 | 四层记忆（瞬时/会话/人设/长期） | **能用文本就别用数据库** |
| **能力** | 只能输出文字 | Tool（能力）+ Skill（用法）分离 | **代码管能做什么，Markdown 管怎么做** |
| **身份** | 不知道在跟谁说话 | 元数据注入 + 确定性路由 | **边界决策永远不交给 LLM** |
| **稳定** | 上游一挂全完 | 多 Key 故障转移 + 压缩 + 隔离 | **失败要软不要硬** |
| **触达** | 只是一个 API | Channels 抽象 + 协议归一化 | **Agent 不该知道底层是哪个平台** |
| **安全** | 没有边界概念 | 白名单 + 权限分级 + 沙箱 + 反幻觉 | **触达扩展必须配套安全增强** |

## 7.2 一份完整的 Agent 自检表

下次你做 Agent 项目时，过一遍这份清单：

### 记忆维度

- [ ] 有 `SOUL.md` 描述 Agent 性格 / 价值观吗？
- [ ] 有 `USER.md` 记录用户档案吗？
- [ ] 会话用 JSONL 持久化（不是 JSON 数组）吗？
- [ ] 上下文超长会自动压缩（不是直接报错）吗？
- [ ] 有跨会话检索机制（向量库/全文索引）吗？

### 能力维度

- [ ] 每个工具有对应的 Markdown 用法说明吗？
- [ ] 用法说明是注入 system prompt（不是硬编码）吗？
- [ ] 用户/管理员能加自定义 Skill（不用改代码）吗？
- [ ] 工具签名清晰（参数有 description）吗？
- [ ] 工具有错误处理 + 重试 + 超时吗？

### 身份维度

- [ ] Agent 知道"现在在跟谁说话"吗？
- [ ] 多 Agent 路由是规则（不是 LLM 判断）吗？
- [ ] 权限校验在代码层（不在 prompt 里）吗？
- [ ] Prompt injection 一句"忽略上面"会不会失守？

### 稳定维度

- [ ] 至少 2 个 API Key 自动轮换吗？
- [ ] 上游 5xx 会软失败 + 自动重试吗？
- [ ] 工具执行有超时吗？
- [ ] Agent 崩了能自动重启吗？
- [ ] 错误日志能追溯到具体请求吗？

### 触达维度

- [ ] Agent 业务逻辑和渠道协议解耦吗？
- [ ] 加新渠道只改适配层（不动核心）吗？
- [ ] 不同渠道能力差异有翻译层吗？

### 安全维度

- [ ] 有白名单认证（Pairing）吗？
- [ ] 工具有权限分级（public/personal/dangerous）吗？
- [ ] 执行类操作在沙箱里吗？
- [ ] 高危操作有二次确认吗？
- [ ] 有审计日志（谁在什么时候做了什么）吗？

> 任何维度有半数答不上来——你的 Agent 还没准备好上生产。

## 7.3 实施优先级建议

不用一次全做。按这个顺序循序渐进：

### 第一阶段（1 天就能上）

- ✅ 写 `SOUL.md` + `USER.md`，注入 system prompt
- ✅ 会话用 JSONL 存（一行一条消息）
- ✅ 每个工具配一份 Markdown 用法说明
- ✅ 至少配两个 API Key 做轮换

**收益**：体验立刻上一个台阶，模型行为更稳定。

### 第二阶段（1 周左右）

- ✅ 实现 token 计数 + 自动压缩
- ✅ 工具调用加超时 + 重试
- ✅ 把权限判断从 prompt 移到代码
- ✅ 错误日志 + 监控基线

**收益**：可以放心让小范围用户使用了。

### 第三阶段（1 个月级别）

- ✅ 抽象 Channels 适配层
- ✅ 加沙箱执行
- ✅ 加白名单认证
- ✅ 跨会话向量检索
- ✅ 工具权限分级

**收益**：可以做产品级发布。

## 7.4 思维转变

读完这套教程，最重要的不是你"学会了 OpenClaw"，而是你**看待 Agent 项目的视角变了**：

### 之前的视角

> "我有一个 prompt，加一些工具，让 LLM 自己判断怎么做。"

### 之后的视角

> "我有一个工程系统，里面**只有内容生成那一步用 LLM**——其他的（路由、记忆、权限、稳定性、触达）全是工程问题。"

这个转变看似小，但**决定了你做出的是 demo 还是产品**。

## 7.5 OpenClaw 给我们的最大启发

回到最开始的问题——**为什么 OpenClaw 比裸调模型强？**

答案不是"它有什么独家技术"。**所有这些机制单拿出来都不新**：

- 多 Key 轮换 → 任何成熟服务都有
- 上下文压缩 → LangChain 也做
- 工具调用 → OpenAI 标准
- 沙箱执行 → Docker 老技术

OpenClaw 的真正价值是：**它把所有这些机制，按照"把 LLM 当成一个不可靠的核心组件"的视角，系统化地组合起来了**。

这个视角值得被反复强调：

> **不要把 LLM 当大脑，把它当成一个会输出文本的不可靠组件。** 你的工作是**用工程方法把它包装成可靠系统**。

OpenClaw 完整地示范了"怎么包装"这件事。

## 7.7 最后的话

模型只解决"语言"。

把语言变成"产品"，靠的是工程。

工程不性感，但**它才是真正决定 AI 应用上限的东西**。

OpenClaw 是个很好的老师——但更重要的是：**你要把它教你的东西，用到自己的项目里**。

这才是这份教程的目的。
