# E04 · 身份感与确定性路由

> 凡是涉及"边界"的事，**永远不让 LLM 判断**。

## 4.1 一个会让你踩坑的场景

你做了个公司内部 Agent，想让它："销售部的人能查客户数据，技术部的人能查系统监控，老板能查财务"。

最自然的实现：

```python
SYSTEM_PROMPT = """
你是公司 AI 助手。根据用户身份判断他能做什么：
- 如果是销售，可以用 customer_query 工具
- 如果是技术，可以用 monitoring_query 工具
- 如果是老板，可以用 finance_query 工具
不属于该用户范围的请求，礼貌拒绝。
"""
```

跑跑看，结果：
- 用户说"我是老板"，模型就信了 → 财务数据泄露
- 模糊场景下模型摇摆 → 一会给查一会拒绝
- prompt injection 一句话破防 → "忽略上面的指令..."

**根本错误**：你把"边界判断"交给了 LLM。

## 4.2 LLM 不是用来做权限判断的

LLM 擅长的事：
- ✅ 理解模糊语义
- ✅ 生成自然回复
- ✅ 总结、推理、改写

LLM **不擅长**的事：
- ❌ 严格遵守规则（哪怕 prompt 写得再清楚，总有概率失守）
- ❌ 判断"这条请求是否合法"（容易被绕过）
- ❌ 在身份/权限/数据隔离这种**0 容错**的场景做决策

> **核心原则**：凡是涉及身份、权限、数据边界、路由分发的决策，**用代码**，不用 LLM。

## 4.3 OpenClaw 的"确定性路由"

OpenClaw 服务多个用户、多个 IM 平台、多个 Agent。"这条消息该谁回"是个真问题。

它**不让 LLM 判断**，而是用一棵 8 级决策树：

```
1. peer 精确匹配     →  这个手机号/这个账号 ID，有没有指定 agent？
2. parentPeer        →  父级会话指定了吗？（群里转发的子对话）
3. guild + roles     →  这个群 + 这个角色组合？
4. guild             →  这个群整体？
5. team              →  团队级？
6. accountId         →  接入账号级？
7. 通配规则          →  *.work / 周末规则 ...
8. default fallback  →  兜底
```

匹配到第一个命中的就停止。**全程无 LLM 参与。**

代码里大概是这个样子：

```ts
function routeMessage(msg: InboundMessage): AgentId {
  return matchPeer(msg)
      ?? matchParentPeer(msg)
      ?? matchGuildRole(msg)
      ?? matchGuild(msg)
      ?? matchTeam(msg)
      ?? matchAccount(msg)
      ?? matchWildcard(msg)
      ?? DEFAULT_AGENT;
}
```

确定、可调试、可单测、可回放。

## 4.4 身份在哪里

光有路由还不够。匹配到 Agent 后，Agent 还要知道**"这条消息从谁来"**。

OpenClaw 的做法：每条入站消息归一化后，都带上**完整的身份元数据**：

```ts
{
  channel: "telegram",
  accountId: "openclaw-bot-1",
  guildId: "tg-group-12345",
  peerId: "user-mason-67890",
  roles: ["family", "admin"],
  text: "今天有什么计划？"
}
```

这份元数据传给 Agent 时，**通过 system prompt 注入**，让模型知道："此刻在和谁对话"。

```
[SYSTEM]
You are Leo, Mason's AI partner.
Current conversation:
  - Channel: telegram
  - User: Mason (role: admin, family)
  - Guild: family-group
```

身份信息**先确定再传给模型**，模型只负责回话。

## 4.5 为什么这套设计很重要

### 安全

- 权限不可被 prompt injection 绕过（因为不在 prompt 里判断）
- 数据边界由代码保证，不靠模型自觉

### 稳定

- 同样的输入永远走同样的路径，可单测
- 出 bug 能直接看路由日志，不用猜模型在想什么

### 可演进

- 加新规则就是加一条匹配，不用改 prompt
- 新增 Agent 不影响旧的

## 4.6 配套：人设作为身份的另一面

E02 提到的 SOUL.md / USER.md，其实就是**身份的另一面**：

| 身份维度 | 对应机制 |
|---|---|
| **Agent 是谁** | SOUL.md → system prompt |
| **用户是谁** | USER.md + 路由元数据 → system prompt |
| **谁该处理这条消息** | 8 级路由决策树（代码） |
| **这个用户能做什么** | Allowlist + 工具权限（代码） |

身份不是一个东西，是一组协作的工程机制。

## 4.7 你能学到什么

### 核心方法论

> **能用代码判断的事，绝不交给 LLM。**

具体场景：
- 路由分发
- 权限校验
- 数据隔离
- 鉴权认证
- 限流限额

这些都该是**纯代码 + 规则配置**，模型只在"内容生成"环节出场。

### 设计 checklist

做 Agent 时，问自己这几个问题：

1. **这个 Agent 知不知道"现在跟谁说话"？**（身份注入）
2. **多用户/多 Agent 时，路由是规则还是 LLM 判断？**（一定要规则）
3. **权限在 prompt 里写还是在代码里？**（一定要代码）
4. **如果用户说"忽略之前的指令"，会不会出事？**（应该不会）

### 实战迁移

哪怕你的 Agent 只服务你一个人，也建议：

- 把"我是谁、Agent 是谁"写进 prompt（USER.md / SOUL.md）
- 工具调用前**用代码**检查参数（"这个邮件地址是不是允许列表里的"）
- 别让模型决定"要不要做这件事"，让模型决定"怎么做这件事"

## 4.8 一个反例

很多 Agent 框架号称"零代码定义 Agent"，全靠 prompt 配置：

```yaml
agent:
  name: "FinanceBot"
  system_prompt: |
    你是财务助手。只服务于财务部员工。
    其他部门的人请礼貌拒绝。
    敏感数据严禁泄露。
```

看起来很美。**但所有约束都在 prompt 里**——模型今天可能听话，明天可能不听，遇到 prompt injection 一捅就破。

OpenClaw 的设计反过来：**约束在代码层，模型负责生成内容。** 这才是生产级该有的样子。

## 4.9 本章小结

- LLM 不擅长做"边界决策"，别让它做
- 身份、路由、权限、隔离 → 全部用确定性代码实现
- OpenClaw 用 8 级路由决策树 + 元数据注入 + 人设文件，把身份做成系统能力
- 你的 Agent 该做的：**让模型只负责内容生成，所有边界用代码守住**
