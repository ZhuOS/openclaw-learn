# E02 · 记忆是工程问题，不是模型问题

> 别再纠结"模型能不能记住"了——它**根本就不能**。这是工程要解决的事。

## 2.1 一个常见的误解

经常有人问："Claude 4 能记住 200K token，是不是能记我说过的所有话？"

**错。** 这是混淆了两件事：

| 概念 | 是什么 | 持续多久 |
|---|---|---|
| **上下文窗口** | 模型这次 API 调用能"看到"的最大文本量 | 只在这一次调用内 |
| **记忆** | Agent 能跨调用、跨会话、跨重启地保留信息 | 由你的工程决定 |

模型本身**没有任何记忆**。每次 API 调用都是无状态的——你不塞进去，它就不知道。

> 所以"让 Agent 有记忆"这件事，**100% 是工程问题**。

## 2.2 OpenClaw 的四层记忆体系

OpenClaw 把记忆拆成 4 层，每层职责不同：

```
┌─────────────────────────────────────────────┐
│ ① 瞬时记忆  =  当前请求的 messages 数组      │  生命周期：1 次 API 调用
│           ↕                                  │
│ ② 会话记忆  =  JSONL 持久化的消息树           │  生命周期：单个对话
│           ↕                                  │
│ ③ 人设记忆  =  SOUL.md / USER.md             │  生命周期：Agent 整个生命
│           ↕                                  │
│ ④ 长期记忆  =  QMD 向量库（跨会话/跨 Agent）  │  生命周期：永久
└─────────────────────────────────────────────┘
```

**关键认知**：这 4 层不是替代关系，是**职责分工**。它们组合起来，才构成"Agent 的完整记忆"。

## 2.3 一层一层看

### ① 瞬时：上下文窗口管理

模型每次调用看到的就是 `messages[]`。看似简单，但有两个工程问题：

**问题 1：超长怎么办？**

200K context 听着大，真实业务里几次工具调用就吃光了（一次浏览器返回的 HTML 可能 50K）。

OpenClaw 的方案：**压缩（compaction）**。
- 监测 token 使用率
- 接近上限时，自动把前面的历史**摘要成短文本**
- 替换原消息，腾出空间继续对话

**问题 2：哪些必须保留？**

不是所有消息都该被压缩。OpenClaw 区分：
- **必留**：system prompt、最近 N 条、当前 tool call 链
- **可压**：早期的对话、已完成的子任务

### ② 会话：JSONL 消息树

很多框架把会话存成 `[]` 数组，OpenClaw 存成**树**：

```
msg_001 (user)
  └─ msg_002 (assistant + tool_call)
       └─ msg_003 (tool_result)
            └─ msg_004 (assistant)
                 ├─ msg_005a (用户分支A：继续问)
                 └─ msg_005b (用户分支B：换话题)
```

每条消息有 `id` 和 `parentId`，天然支持：
- **会话回溯**（回到任意 checkpoint 重新生成）
- **分支探索**（同一个起点，试两种方向）
- **删除特定分支**而不影响其他

**为什么用树而不是数组？** 因为真实使用中你**经常想回退、重试、换个方向**。线性数组做不到。

### ③ 人设：Markdown 即配置

这是最反直觉、也最巧妙的设计：

> Agent 的"性格 / 用户档案 / 项目背景"，都用纯 Markdown 文件管理。

```
~/.openclaw/
  ├── SOUL.md      # Agent 自己的性格、说话风格、价值观
  ├── USER.md      # 用户档案：名字、偏好、家庭情况
  ├── AGENTS.md    # 这个 Agent 的能力边界、行为约束
```

**Agent 启动时，把这几个文件的内容拼进 system prompt**——它每次"醒来"都有完整的身份感和用户认知。

**为什么这么做巧妙？**

1. **可读可改**：你能直接打开文件看 Agent 是什么人，想改就改
2. **版本可控**：可以 git 管理，回看变化
3. **跨工具迁移**：换个 Agent 框架，文件搬过去就行
4. **AI 自己也能改**：Agent 可以更新自己的 SOUL.md（"我学到了用户喜欢直接表达"）

> **方法论**：能用纯文本管理的状态，就别用数据库。可读性 > 一切。

### ④ 长期：QMD 向量检索

跨会话的事实型记忆（"用户老婆叫什么"、"上次部署用的什么命令"），存在 **QMD（Qdrant-backed Markdown Database）** 里：

```
向量库         |  关联的 Markdown 文件
embedding_001  →  /memories/family/wife.md
embedding_002  →  /memories/work/deploy-history.md
embedding_003  →  /memories/projects/openclaw-learn.md
```

Agent 调用 `memory_search("我老婆生日")` 工具时：
1. 把查询转 embedding
2. Qdrant 找出最相似的几条
3. 读对应 Markdown 文件返回

**关键设计**：向量只是**索引**，真正的内容存在可读的 Markdown 文件里。这意味着——

- 检索失败时你能直接 `grep` 找内容
- 内容可以被多个 Agent 共享（`extraCollections` 配置只读访问）
- 清理或迁移时，操作的是文件，不是向量库

## 2.4 四层怎么协作

举个真实例子：你问 Agent「上次咱们讨论的那个 OpenClaw 教程进度怎么样？」

| 步骤 | 用到哪层 | 做什么 |
|---|---|---|
| 1. Agent 启动 | ③ 人设 | 读 SOUL.md/USER.md，知道"我是 Leo，用户是 Mason" |
| 2. 收到问题 | ① 瞬时 | 当前 messages 加上系统提示和这条问题 |
| 3. 触发检索 | ④ 长期 | `memory_search("OpenClaw 教程")` 找到相关历史 |
| 4. 拼装上下文 | ② 会话 | 加载这次对话之前的几轮交流 |
| 5. 调模型 | ① 瞬时 | 把 ②③④ 的内容一并塞进 messages |
| 6. 输出 | - | "Ch05 上周写完了，今天部署到了 GitHub Pages..." |

**没有任何一层是"模型自己做的"。** 全部是工程层在调度。

## 2.5 你能从这一层学到什么

### 设计原则

1. **分层** — 把记忆按生命周期分层，每层只解决一类问题，不要混
2. **可读** — 能用纯文本就别用数据库；调试时眼睛能看的胜过一切
3. **可控** — 让记忆是显式的，不是黑盒；什么时候加什么时候删要清晰
4. **可恢复** — 树状结构 + JSONL 持久化，意味着**没有"丢消息"这件事**

### 直接可迁移的方法

如果你现在在做自己的 Agent，最低成本的迁移是这几条：

- **一个 `~/.myagent/SOUL.md`** — Agent 性格写进去，比硬编码 system prompt 强 100 倍
- **一个 `USER.md`** — 用户偏好和事实写进去，每次拼进 prompt
- **JSONL 存对话** — 一行一条消息，带 `id/parentId`，比 JSON 数组好维护
- **加一个 `memory_search` 工具** — 哪怕只用文件 grep 实现，先有再说

这些**今天下午就能加上**，立刻让你的 Agent 体验上一个台阶。

## 2.6 反例：不分层的下场

我见过的常见错误：

```python
context = previous_summary + last_5_messages + user_facts + agent_personality + current_input
response = llm(context)
```

把所有东西揉成一个字符串塞进 prompt。短期能跑，但：
- 想改"用户事实"，得在拼字符串的代码里改
- 想做会话回溯，没结构
- 想跨会话检索，没索引
- 想给 Agent 换个性格，得改代码

**分层的本质是"关注点分离"**——做 Agent 和做软件是一回事。

## 2.7 本章小结

- 记忆是工程问题，**不要指望模型本身**
- OpenClaw 用四层（瞬时/会话/人设/长期）拆开管理，职责清晰
- 关键设计：纯文本管理、JSONL 树、向量索引指向 Markdown
- 你的 Agent 立刻能用：`SOUL.md` + `USER.md` + JSONL 会话 + 简单 memory_search
