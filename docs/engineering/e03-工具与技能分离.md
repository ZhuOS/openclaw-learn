# E03 · 工具与技能的分离设计

> 模型"会调工具"和"知道怎么用工具"，是两件事。

## 3.1 一个被忽略的事实

你给 LLM 一个 `send_email` 工具，它一定会用对吗？

不一定。它可能：
- 在该用的时候不用（"我建议你发个邮件..." 而不是真的发）
- 在不该用的时候滥用（每个回答都想发邮件）
- 调用参数错（收件人填成正文）
- 调用顺序错（先发邮件再问"要发给谁"）

为什么？因为 **function calling 给的只是「工具的签名」**——名字、参数、返回类型。它**没告诉模型**：

- 什么场景下该用这个工具
- 用之前要不要确认
- 失败了怎么办
- 跟其他工具怎么配合

这些是**用法**，不是签名。模型本身不知道。

## 3.2 OpenClaw 的三层分离

OpenClaw 把"扩展能力"拆成三层：

```
┌──────────────────────────────────────────────┐
│  Tools    =  能力本身（函数签名 + 实际代码）   │
│  Skills   =  用法说明（Markdown 注入 prompt） │
│  Plugins  =  打包分发（npm 包 + 元数据）      │
└──────────────────────────────────────────────┘
```

最重要的是 **Tools 和 Skills 的分离**。这是这一章的核心。

## 3.3 Tools：工具的"硬件"

Tool 就是一个标准的 function：

```ts
{
  name: "send_telegram_message",
  description: "Send a message to a Telegram chat",
  parameters: {
    chatId: { type: "string" },
    text: { type: "string" },
  },
  execute: async ({ chatId, text }) => {
    return await telegramApi.send(chatId, text);
  }
}
```

模型看到这个签名，能判断"这是个发 Telegram 消息的工具"。但**它不知道**：

- 怎么获取 chatId？
- 长消息要不要分段？
- 失败重试机制？
- 什么时候用 markdown，什么时候用 plain text？

这些都得**额外告诉它**。

## 3.4 Skills：工具的"使用说明书"

Skill 是一个 Markdown 文件，**注入到 system prompt**。它解释**怎么用工具**：

```markdown
# Skill: telegram-messaging

## 何时使用
当用户明确要求"发送 Telegram 消息"或"通知 XX"时使用。
不要在以下场景使用：
- 用户只是在聊天里提到"Telegram"，不是要求发送
- 没有明确收件人时（先问，不要猜）

## 使用步骤
1. 确认收件人：使用 `get_telegram_contacts` 找到 chatId
2. 检查消息长度：超过 4000 字符要分段
3. 发送：调用 `send_telegram_message`
4. 出错处理：如果返回 429，等待 5 秒重试一次

## 格式约定
- 私聊用纯文本
- 群聊用 markdown，加粗关键词
- 超长内容贴成代码块

## 反例
❌ 不要：用户问"今天天气如何"，你说"我已经发到 Telegram 了"
✅ 应该：用户说"把天气预报发给我老婆"，先确认 chatId，再发送
```

这份 Markdown 在 Agent 启动时被读出来，**作为 system prompt 的一部分**塞进去。模型每次回答前都会"看到"这份说明书。

## 3.5 为什么这种分离是天才设计

### 优点 1：解耦更新

- 改用法（什么场景下用、怎么配合）→ 改 Skill Markdown，**不用动代码**
- 改实现（API 升级、加重试逻辑）→ 改 Tool 代码，**不用动 prompt**

### 优点 2：可读 + 可审计

你能直接打开 SKILL.md 看"这个 Agent 在什么场景下会怎么做"。**这是黑盒变白盒。**

对比：很多框架把"使用规则"塞进硬编码的 system prompt 字符串里，**改一次得改源码、重启服务**。

### 优点 3：用户可控

普通用户不会写代码，但**会写 Markdown**。

OpenClaw 让用户**自己写 Skill** 来定制 Agent 行为：
- "我希望发邮件前总是先给我看草稿" → 加一条到 Skill
- "周末别给我发工作消息" → 加一条到 Skill

不用改代码、不用改框架，**纯文本即配置**。

### 优点 4：模型友好

LLM 训练时见过大量 Markdown 文档，**Markdown 是它最熟悉的"指令格式"**。比 JSON schema、比硬塞的字符串都好理解。

## 3.6 反幻觉的关键：Skills 是"先验"

模型有个老毛病：**不知道有什么工具的时候，会编一个**。

```
用户：帮我把这个文件转成 PDF
模型：好的，我使用 convert_to_pdf 工具...
（实际：你根本没注册这个工具，但模型自信地"假装调用"了）
```

OpenClaw 的 Skills 机制是天然的反幻觉：
- 每个工具都有对应的 Skill
- Skill 里**明确声明**"如果用户要求 X 但你没有对应工具，告诉他做不到"
- 工具列表 + 用法说明都在 prompt 里，模型没有"自己编"的空间

## 3.7 Plugins：打包让生态可分发

Plugins 是 Tools + Skills 的打包形式：

```
my-weather-plugin/
  ├── package.json        # npm 包元数据
  ├── tools/
  │   └── get-weather.ts  # 一个或多个 Tool
  ├── skills/
  │   └── weather.md      # 配套的 Skill
  └── plugin.json         # 给 OpenClaw 用的清单
```

发布到 ClawHub（OpenClaw 的插件市场），其他人 `openclaw plugin install weather` 就能用。

**这一层的意义不在于技术新意，而在于"生态可能"**：能力可以被打包、分发、组合。

## 3.8 你能学到什么

### 核心方法论

> **能力（Tool）和用法（Skill）必须分离。**

代码里只放"能做什么"，Markdown 里写"什么时候做、怎么做"。

### 实战迁移

如果你正在做 Agent，现在就可以：

1. **每个工具配一个 SKILL.md** — 说清楚使用场景、步骤、反例
2. **启动时把所有 SKILL.md 拼进 system prompt** — 让模型"看到"用法
3. **让用户能加自己的 SKILL.md** — 自定义行为不用改代码
4. **用 Markdown 写规则，不用 JSON / 代码字符串** — 模型最懂 Markdown

### 一个常见的反模式

```python
SYSTEM_PROMPT = """
You are a helpful assistant. You have access to these tools:
- send_email: 用法是先获取邮箱地址，然后...
- query_database: 用法是先连接，然后...
- ...（300 行）
"""
```

把所有用法塞进一个巨大字符串。**坏处**：
- 改一条规则要改源码
- 用户看不到也改不动
- 长 prompt 模型容易遗忘

**正确姿势**：拆成多个 SKILL.md，按需加载。

## 3.9 本章小结

- Tool 是能力（"能做什么"），Skill 是用法（"怎么做"）
- 分离的好处：解耦更新、可读可审计、用户可控、反幻觉
- Plugins 是 Tools + Skills 的打包，让生态可分发
- 你的 Agent 可立即学的：**每个工具配一份 Markdown 用法说明**
