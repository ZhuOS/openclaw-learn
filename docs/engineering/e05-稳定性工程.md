# E05 · 稳定性工程：让 Agent "不断线"

> Demo 跑通是一回事，跑一个月不挂是另一回事。

## 5.1 真实世界很残酷

你的 Agent 跑通了 demo，兴致勃勃部署上线。一周后回头看：

- 周二凌晨 OpenAI 限速，所有用户收到 429
- 周三某次对话上下文涨到 200K，模型直接报错
- 周五 Anthropic 维护，服务全断了 4 小时
- 周末有人发了张超大图，Pi Agent 卡死
- 周一某个 tool 调用 timeout，整个对话挂起

这些**全是工程问题**，模型本身一行代码都不能帮你。

OpenClaw 的稳定性工程，就是为了**让上面这些场景都"自动恢复"**。

## 5.2 三个关键机制

OpenClaw 在稳定性上做了三件事：

```
① 多 Key 故障转移   →  上游限速/挂掉时自动切换
② 上下文压缩防爆    →  长对话不崩盘
③ 工具执行隔离      →  单个工具挂掉不连累整个 Agent
```

一个一个看。

## 5.3 多 Key 故障转移（authProfile）

LLM API 提供商不靠谱是日常：限速、限流、运维窗口、key 被风控。

OpenClaw 的方案：**每个 Agent 配多个 authProfile，自动轮换 + 冷却**。

配置长这样：

```json
{
  "agentId": "leo",
  "authProfiles": [
    { "name": "anthropic-main", "apiKey": "...", "priority": 1 },
    { "name": "anthropic-backup", "apiKey": "...", "priority": 2 },
    { "name": "openai-fallback", "apiKey": "...", "priority": 3, "model": "gpt-4o" }
  ],
  "cooldownMs": 60000
}
```

行为：

```
请求来 → 用 priority 1 的 anthropic-main
         ↓ 返回 429 / 5xx
         → 标记为冷却 60 秒
         → 立即用 priority 2 的 anthropic-backup
            ↓ 也挂了？
            → 用 priority 3 的 openai-fallback（不同厂商）
```

**关键设计**：
- 冷却是临时的，过了时间会自动恢复
- 优先级让你能定义"主力 + 备份"
- 跨厂商兜底，避免单厂商全挂时彻底断线
- 整个过程**对用户完全透明**——他不知道 Key 切换过

> **方法论**：任何外部依赖都该有"故障转移 + 冷却 + 自愈"机制。

## 5.4 上下文压缩防爆（compaction）

E02 提过压缩，这里讲它的稳定性意义。

**问题**：长对话会触发两类崩盘
- **硬崩**：超过模型 context limit，直接报错
- **软崩**：context 太长，token 成本爆炸 + 响应变慢 + 模型遗忘前文

OpenClaw 的对策：**compaction-safeguard 钩子**，在每次调用前检查并压缩。

伪代码：

```ts
beforeRequest(messages) {
  if (countTokens(messages) > THRESHOLD) {
    const old = messages.slice(0, -KEEP_RECENT);
    const summary = await summarize(old);
    return [
      systemPrompt,
      { role: "system", content: `[Earlier summary]: ${summary}` },
      ...messages.slice(-KEEP_RECENT)
    ];
  }
  return messages;
}
```

**保留**：
- system prompt（必须）
- 最近 N 条（保上下文连续性）
- 当前 tool call 链（不能截断未完成的工具调用）

**压缩**：
- 早期对话 → 摘要成短文本
- 完成的子任务 → 摘要成结果

**效果**：你的 Agent 可以聊 8 小时不爆炸，token 成本可控，体验连续。

> **方法论**：不要假设输入长度。任何会无限增长的状态，都要有**淘汰机制**。

## 5.5 工具执行隔离（context-pruning）

工具调用的返回值有时候大得离谱：
- 浏览器返回 100K HTML
- 数据库查询返回 10000 行
- 文件读取一个 5MB 的日志

如果不管，下一次对话这些都还在 context 里。

OpenClaw 的方案：**context-pruning 钩子**，工具结果有"生存期"——

- 当前轮次：完整保留
- 跨过若干轮：自动**修剪**或**摘要**
- 旧的工具结果替换为 `[tool result pruned, X bytes]`

模型还能"看到这里曾经有个工具结果"，但不再占用 token。

## 5.6 沙箱隔离

工具执行也要隔离：

```
Agent 主进程
   │
   ├─ Tool A：读文件 → 普通函数调用
   ├─ Tool B：执行代码 → Docker 沙箱
   └─ Tool C：浏览器 → 子进程 + 超时控制
```

某个 Tool 卡住、崩了、产生死循环，**不会拖死 Agent 主进程**。

这是从操作系统借来的思想：**故障域隔离**。

## 5.7 健康检查与自愈

OpenClaw Gateway 还有完整的健康监测：

- **Heartbeat**：每个 Agent 定期上报存活
- **Presence**：Channels 状态汇总（Telegram bot 是否在线？）
- **Failure detection**：连续失败 N 次自动降级
- **Restart policy**：crash 后自动重启（基于 systemd / launchd）

这些不性感，但**生产环境缺一不可**。

## 5.8 你能学到什么

### 核心方法论

> **稳定性不是模型的事，是系统的事。**

任何 LLM Agent 要上生产，必须想清楚：

1. **上游挂了怎么办？**（多 Key、跨厂商兜底）
2. **上下文涨到爆怎么办？**（自动压缩）
3. **工具执行卡死怎么办？**（隔离 + 超时）
4. **Agent 自己崩了怎么办？**（监控 + 自愈）

### 立刻可用的清单

哪怕你的 Agent 只是个个人项目，建议加上：

- [ ] **至少 2 个 API Key 轮换** — 当一个挂了，对话不中断
- [ ] **token 计数 + 阈值压缩** — 别等爆了才处理
- [ ] **每个工具调用加超时** — 默认 30s，不能让一个 hang 拖死 Agent
- [ ] **错误日志 + 监控** — 知道哪里失败了、多久一次

### 一个常被忽略的点

> **失败要"软"**，不要"硬"。

❌ 硬失败：抛异常 → 整个对话崩 → 用户看到 500 错误
✅ 软失败：捕获 → 降级回复"这个工具暂时不可用，我换个方式" → 继续对话

这个"软"字差异，就是 demo 和产品的差距。

## 5.9 一个对比

某个常见的 Agent 实现（典型反例）：

```python
def chat(message):
    response = openai.chat.completions.create(
        model="gpt-4",
        messages=history + [{"role": "user", "content": message}]
    )
    history.append({"role": "user", "content": message})
    history.append({"role": "assistant", "content": response.choices[0].message.content})
    return response.choices[0].message.content
```

跑得通。**但**：
- API 挂了 → 直接抛异常
- history 无限增长 → 迟早爆 context
- 没有重试、没有降级、没有日志

OpenClaw 的版本（思路）：

```ts
async function chat(message) {
  const profile = pickHealthyProfile();
  const ctx = await compactIfNeeded(history);

  try {
    return await callWithTimeout(profile, ctx, message);
  } catch (e) {
    markUnhealthy(profile);
    return chat(message);  // 自动用下一个 profile
  }
}
```

差距就在这里。

## 5.10 本章小结

- 稳定性是工程问题，模型本身解决不了
- OpenClaw 三件套：多 Key 故障转移 + 上下文压缩 + 工具隔离
- 配套：健康检查、监控、自愈、软失败
- 你的 Agent 要上生产，先答这四题：上游挂了？context 爆了？工具卡了？Agent 崩了？
