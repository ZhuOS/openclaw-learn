# E06 · 触达扩展与安全边界

> 让 Agent **能去到用户在的地方**，并且**敢真的做事**——这两件事是一对。

## 6.1 一个被低估的维度

很多人做 Agent 时，默认就是"做个聊天界面"。

但真实需求是这样的：

> "我希望我的 AI 助手在我用 Telegram 的时候能用 Telegram，用微信的时候能用微信，开会时能在会议群里待命。"
> "我老婆不会用聊天 App，但她会用 iMessage——能不能让她也用？"
> "我希望它能主动给我发消息，不只是回答问题。"

这些都是"触达"问题。

但触达扩大后，**风险也跟着扩大**：
- Telegram 群里有外人，怎么保证敏感数据不泄露？
- iMessage 被盗号了，怎么防止 Agent 被恶意指使？
- Agent 会执行代码、写文件，怎么保证它不删错文件？

**触达和安全是一对**——你想去更多地方，就必须建更严的边界。

## 6.2 触达：Channels 抽象层

OpenClaw 把"渠道"做成可插拔的抽象层：

```
Telegram | WhatsApp | Discord | iMessage | Slack | Web | Email | ...
   │         │          │          │         │      │     │
   └─────────┴──────────┴──────────┴─────────┴──────┴─────┘
                          ▼
              ┌──────────────────────┐
              │  Channels 抽象层      │  ← 协议归一化、能力翻译、配对认证
              └──────────────────────┘
                          ▼
                   统一的事件流
                          ▼
                       Agent
```

每个渠道做三件事：

### ① 协议归一化

Telegram 的 webhook 格式 vs WhatsApp 的 vs iMessage 的——**完全不一样**。Channels 把它们都翻译成统一的 `InboundMessage` 格式：

```ts
{
  channel: "telegram",
  accountId: "...",
  guildId: "...",
  peerId: "...",
  text: "今天天气如何",
  attachments: [],
  timestamp: 1730000000000
}
```

Agent 只看到归一化后的事件，**不用关心底层是哪个平台**。

### ② 能力翻译

不同平台支持的能力不一样：
- Telegram：支持 markdown、行内键盘、文件
- WhatsApp：支持媒体、quick replies、不支持 markdown
- iMessage：纯文本 + 附件，不支持交互按钮

Channels 层负责"翻译"：
- Agent 输出 markdown → Telegram 直接发；WhatsApp 转纯文本
- Agent 给按钮选项 → Telegram 显示按钮；iMessage 显示编号文字"回复 1/2/3"

**Agent 不用为每个平台写不同代码**——它只负责输出"语义上的回复"，渠道层负责"格式适配"。

### ③ 配对（Pairing）

谁能用这个 Agent？OpenClaw 用 **Pairing 机制**：

1. 你启动 Agent，得到一个一次性的 pairing code
2. 用户在他的 IM 里发 `/pair ABC123` 给 bot
3. Pairing 成功 → 这个用户的 peerId 加入 Allowlist
4. 之后他就能用了

这避免了**"Bot 暴露在公网，谁都能调用"**的问题。

> **方法论**：增加触达 = 增加攻击面。每个新渠道都要有自己的认证机制。

## 6.3 安全：四层防护

OpenClaw 在安全上做了 4 层：

```
┌──────────────────────────────────────┐
│ ① Allowlist：白名单认证               │  谁能用
│ ② 工具权限：四档分级                  │  能用什么工具
│ ③ Sandbox：执行隔离                   │  执行不会越界
│ ④ Skills 反幻觉：明确声明能力边界     │  模型不乱编
└──────────────────────────────────────┘
```

### ① Allowlist

每个 Agent 维护自己的 peer 白名单：

```json
{
  "agentId": "leo",
  "allowedPeers": [
    { "channel": "telegram", "peerId": "user-mason" },
    { "channel": "imessage", "peerId": "user-mason-wife", "scope": "family-only" }
  ]
}
```

不在白名单的，直接拒绝，**不进入 Agent 处理流程**。

### ② 工具权限四档

不是所有用户都该用所有工具。OpenClaw 把工具调用权限分四档：

| 档位 | 含义 | 例子 |
|---|---|---|
| **public** | 任何认证用户可用 | 查天气、查日历 |
| **personal** | 主人专用 | 读私人文件、记笔记 |
| **system** | 受信任设备 | 改配置、重启服务 |
| **dangerous** | 需二次确认 | 删文件、转账、发邮件 |

这套权限**配置在工具元数据里**，调用前由代码检查，**不靠 prompt 约束**。

### ③ Sandbox

工具执行（特别是 `exec`、写文件、运行代码）默认在 Docker 沙箱里跑：

```
Agent 进程
    ↓ 调用 exec("rm -rf /")
工具调度器
    ↓ 在沙箱里执行
Docker 容器（隔离的文件系统、隔离的网络）
    ✓ 影响只局限在容器内
```

即使 Agent 被 prompt injection 攻击，让它执行恶意代码——**伤害也被沙箱拦住**。

### ④ Skills 反幻觉

E03 讲过 Skills。这里讲它的安全意义：

每个 Skill 里**明确声明**能力边界：

```markdown
# Skill: file-management

## 能做什么
- 读取 ~/projects/ 目录下的文件
- 写入 ~/scratch/ 目录

## 不能做什么
- ❌ 读写 ~/ 之外的路径
- ❌ 删除任何文件（如需删除，告诉用户手动操作）
- ❌ 执行 .sh / .py 脚本（用专门的 exec 工具）
```

这层防御是给模型看的——让它**主动拒绝**不该做的事，作为"代码硬约束"之外的软约束。

## 6.4 触达和安全的互锁设计

OpenClaw 的精妙之处：触达和安全是**互锁的**。

| 触达扩展 | 对应的安全增强 |
|---|---|
| 加新渠道 | 强制要求 Pairing 认证 |
| 加新用户 | 默认 public 权限，敏感工具需手动加白 |
| 加新工具 | 必须声明权限档位 + 配 Skill |
| 加 dangerous 工具 | 必须支持二次确认 |

**没法只扩展一边**。这种设计强制你想完整，避免"上线后才发现安全没做"。

## 6.5 你能学到什么

### 核心方法论

1. **触达和安全是同一件事的两面** — 想多触达，必须多安全；想多安全，先认清触达边界
2. **认证用代码，不用 prompt** — 谁能调、能调什么，全在代码层
3. **执行用沙箱** — 哪怕你信任 Agent，也别信任它生成的代码
4. **能力声明是双重的** — 代码层硬约束 + Skills 软约束，互为保险

### 实战 checklist

做 Agent 时，问自己：

- [ ] **认证**：谁能用这个 Agent？怎么验证身份？
- [ ] **工具分级**：每个工具是 public 还是 dangerous？
- [ ] **执行隔离**：会执行代码吗？有沙箱吗？
- [ ] **二次确认**：高危操作（删除/支付/外发）有 confirm 步骤吗？
- [ ] **审计日志**：谁在什么时候调用了什么？能查得到吗？
- [ ] **Prompt injection 防御**：用户能否通过对话绕过权限？

如果有任何一项答不上来，**先别上线**。

## 6.6 一个真实代价

很多人做 Agent 时跳过安全，理由是"反正只我自己用"。

直到：
- 把 Agent 链接到 GitHub，结果它把私有 repo 内容贴到群里
- 给 Agent 文件读权限，结果某次 prompt 让它读了 `.ssh/id_rsa`
- Telegram bot 被人加了，"我是管理员"一句话拿到所有数据

**这些都不是模型 bug，是工程没做。**

OpenClaw 不是为了限制你而做这些，是为了让你**敢真的把 Agent 用起来**。

## 6.7 本章小结

- 触达扩展（Channels）和安全边界（Allowlist + 权限 + 沙箱）是一对
- OpenClaw 用 Channels 抽象层做协议归一化和能力翻译，让 Agent 跨平台
- 安全用四层叠加：白名单 + 工具权限 + 沙箱 + Skills 声明
- 你的 Agent 要上线前，过一遍六项 checklist——任何一项答不上来都别上
