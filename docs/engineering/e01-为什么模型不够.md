# E01 · 为什么裸调模型不够用

> 一切的起点：搞清楚"模型给你的"和"你需要的"之间，差了什么。

## 1.1 一个常见的错觉

很多人第一次写 Agent 的时候，思路是这样的：

```python
client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "帮我整理今天的会议纪要"}]
)
print(response.choices[0].message.content)
```

跑通了。觉得"AI Agent 也就这么回事嘛"。

然后你想做点真正有用的东西：

> "我想让 AI 助手帮我管理日程、记住我的偏好、能调用我的内部工具、在 Telegram 和飞书都能用、家人也能用但只能问有限的事..."

这时候你开始懵了。**每个需求都不是模型本身能解决的。**

## 1.2 模型只解决"语言"

让我们诚实地看一眼：当你调用一次 LLM API，模型实际**给了你什么**？

```
输入：messages 数组（一段对话历史）
输出：一段文字 / 一个 tool_call JSON
```

就这两件事。**它不知道**：

- 你是谁（每次请求都是一张白纸）
- 上次聊了什么（除非你自己塞进 messages）
- 它自己的"性格"（除非 system prompt 里有）
- 现在能做什么（除非工具列表里写了）
- 这条消息从哪来、该不该回（它不管，给什么回什么）

模型只解决一件事：**给定一段文字上下文，输出符合人类期待的下一段文字。**

仅此而已。

## 1.3 真实产品需要的能力清单

但你想做的产品，需要的是这些：

| 维度 | 需求 | 模型能给吗 |
|---|---|---|
| 记忆 | 跨会话记住用户偏好、项目历史 | ❌ 完全无 |
| 身份 | 自己叫什么、性格是什么、用户是谁 | ❌ 每次重启即遗忘 |
| 工具 | 真的能写文件、发消息、查数据 | ⚠️ 只能输出"想调用什么"的 JSON，执行得你自己来 |
| 触达 | 在多个 IM / 平台都能用 | ❌ 只是个 API |
| 路由 | 多个 Agent 时该谁处理 | ⚠️ 让模型自己判断会不稳定 |
| 稳定 | API 限速 / Key 失效不断线 | ❌ 完全无 |
| 安全 | 谁能调它、能调到什么程度 | ❌ 完全无 |
| 协作 | 多用户、多设备访问同一个 Agent | ❌ 完全无 |

模型给你 1 分，剩下的 9 分，**全是工程**。

## 1.4 这就是 OpenClaw 解决的问题

OpenClaw 不是"另一个 Agent 框架"，它是**对"LLM 工程化"这个命题的一份完整答案**：

> 它把模型作为**最后一公里**，前面九公里全部用工程方法补齐。

具体来说，它在 6 个维度上做了系统化补全（每个维度后面会单独讲一章）：

```
模型给你的：       一次次聪明的语言输出
        ↓
OpenClaw 加上：
  ① 记忆体系       让 Agent 有"昨天"
  ② 工具与技能     让模型"会做事 + 知道怎么做"
  ③ 身份与路由     让 Agent "知道自己是谁、为谁服务"
  ④ 稳定性         让 Agent "不断线、能恢复"
  ⑤ 多渠道触达     让 Agent "无处不在"
  ⑥ 安全边界       让 Agent "敢做事、做得安全"
        ↓
你拿到的：         一个真正可用的生产力 Agent
```

## 1.5 为什么这件事值得专门讲

你可能会想：「这些不就是工程常识吗？」

**对，但常识不等于做得对。**

很多人做 Agent 时的真实状态是：
- **记忆**靠 prompt 里 `previous_summary: "..."` 这种贴膏药
- **身份**靠每次硬编码 system prompt
- **工具**让 LLM 自己决定调什么，结果各种幻觉
- **路由**多 Agent 之间用 LLM 判断"该谁处理"，结果不稳定
- **稳定**Key 挂了就挂了，自己重启
- **安全**没考虑过，反正 demo 不对外

每一个细节都不难，但**全部做对、且做得清晰可维护**——这才是 OpenClaw 的真正价值。

它给我们示范了：**生产级 LLM Agent 应该在每个维度上做到什么程度。**

## 1.6 你能从这个教程拿走什么

读完这 7 章，你应该能：

- ✅ **诊断**：拿到任何一个 Agent 项目，能快速识别"它在哪个维度做得好、哪个维度凑合"
- ✅ **设计**：自己做 Agent 时，能列出完整的工程清单，不漏维度
- ✅ **判断**：评估各种 Agent 框架/产品（LangChain、Dify、Coze...）时，知道**真正该看什么**

而不仅仅是"会用 OpenClaw"。

## 1.7 本章小结

- 模型只解决"语言"，工程解决"产品"
- 真实产品需要 6+ 个维度的工程能力，模型只给你最后 1 个
- OpenClaw 的价值是**把这套补全方法系统化、可复用化**
