# 工程化篇 · 序言

> 为什么模型很强，但你拿它干活总差点意思？

## 你的困惑可能是

- 用 GPT/Claude 单聊很惊艳，但想让它**持续帮你做事**，每次都要重新交代背景，烦死了
- 写个简单脚本调 OpenAI API，能跑，但**接到真实业务里**——稳定性、权限、协作、记忆——一堆问题
- 看了一堆 Agent 框架（LangChain/AutoGen/CrewAI），感觉都是「玩具」，离生产可用差很远
- 知道 prompt engineering、知道 RAG、知道 function calling，但拼起来总像散件，**没法做出一个真正"有用"的产品**

这一切的根源是同一个事实：

> **模型本身只解决"语言"，不解决"工程"。**

模型给你的是一次次聪明的输出，但要把这些输出变成**长期、稳定、有身份、有记忆、能执行、可协作**的产品级能力，**全靠工程层的补全**。

## OpenClaw 给我们看了什么

OpenClaw 这个项目的真正价值，**不是它能用**，而是它**示范了一套完整的工程方法论**：当你想让 LLM 真正成为生产力工具时，**到底要解决哪些问题、怎么解决**。

这套方法论，可以直接搬到你自己的 Agent / 工具 / 产品里。

## 这个教程要做什么

不读源码。不讲实现细节。我们要做的是：

把 OpenClaw 的工程设计**抽象成 6 条可复用的方法论**，每条都讲清楚：

1. **模型本身的短板是什么**（你不解决就一定踩坑）
2. **OpenClaw 怎么补**（思路，不是代码）
3. **你能学到什么**（怎么用到自己的项目里）

读完之后，你应该获得的是**"如何把 LLM 工程化"的一套思考框架**，而不是 OpenClaw 的使用知识。
