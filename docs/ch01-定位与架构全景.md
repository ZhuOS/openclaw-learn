# Ch01 · 项目定位与架构全景

> 读源码之前，先搞清楚："这东西到底是什么？它想解决谁的什么问题？"

## 1.1 OpenClaw 是什么？

一句话定义：**自托管的、多渠道的 AI Agent 网关（Gateway）**。

拆开来看：

- **自托管**（self-hosted）：跑在你自己的机器/服务器上，配置和数据都在 `~/.openclaw/`，不依赖厂商云
- **多渠道**（multi-channel）：Telegram / WhatsApp / Discord / iMessage / Slack 等 20+ 消息平台同时接入同一个"大脑"
- **AI Agent 网关**：不只是 chatbot，而是内嵌了带工具调用能力的编码智能体（基于 `pi-*` SDK），具备代码执行、浏览器、文件读写等能力

### 它不是什么

| 对比对象 | 区别 |
|---------|------|
| LangChain / LlamaIndex | 那是 Agent 开发框架，OpenClaw 是可直接跑的产品 |
| Dify / Coze | 那是可视化编排平台（SaaS 为主），OpenClaw 是本地 CLI + 后台服务 |
| Claude Code / Cursor Agent | 那是 IDE 里的编码 Agent，OpenClaw 把编码 Agent 搬到了 IM 里 |
| Botpress / Rasa | 那是传统 NLU chatbot，OpenClaw 原生围绕 LLM + tool calling |

## 1.2 核心设计理念

读任何一个项目，先要领会作者的"三板斧"式设计偏好。OpenClaw 的核心哲学：

### ① Gateway-as-Single-Source-of-Truth

所有渠道、所有 Agent、所有会话统统收敛到一个 Gateway 进程。

```
多个 IM 入口 ────┐
多个 Agent  ────┼──►  Gateway（唯一的控制面）
多个节点设备 ────┘         │
                          ├── 会话存储
                          ├── 路由决策
                          ├── 配置中心
                          └── 协议对外暴露（HTTP / WS）
```

这个决定带来的结果是：**代码里到处都能看到 Gateway 这个词**，几乎所有关键状态都绕不开它。

### ② Embedded Pi，不走子进程

作者选择把 Agent SDK（`@mariozechner/pi-*`）**直接 import 进来**，不 spawn 子进程、不走 RPC：

```ts
// 伪代码，实际位置在 packages/agent/src/runtime.ts 附近
import { createAgentSession } from "@mariozechner/pi-agent-core";

const session = createAgentSession({
  workspaceDir,
  tools: [...builtinTools, ...channelTools, ...pluginTools],
  systemPrompt: renderSoulPrompt(channel, sender),
});
```

**为什么重要**：你读代码时不会被 IPC / JSON-RPC 搞晕，所有调用链是**同进程同栈追踪**的。这极大降低了源码阅读难度。

### ③ 确定性路由，不让 LLM 判断"这消息谁处理"

"这条 WhatsApp 消息该由哪个 Agent 回复？"——OpenClaw 的答案是**规则匹配**，不交给 LLM：

```
peer 精确匹配 → parentPeer → guild+roles → guild → team
             → accountId → 通配 → default fallback
```

这个决策树在 Ch08 会深入，但**设计哲学**你要先记住：**凡是影响身份、权限、数据边界的决策，都不让 LLM 参与**。

### ④ 能力扩展三层分离

```
Tools   = 能力（函数签名、实际代码）
Skills  = 用法（Markdown 注入到 system prompt）
Plugins = 打包（npm 包 + 元数据）
```

这三层解耦的好处在 Ch07 你会真切感受到——你可以只加一个 Skill 就改变 Agent 行为，**不用动任何代码**。

```
┌──────────────────────────────────────────────────────────────────┐
│                         用户入口层                                 │

                          │ 消息（统一事件格式）

┌──────────────────────────────────────────────────────────────────┐
│                   Channels 渠道适配层                              │
│  每个渠道一个适配器：协议转换 + 能力降级 + 配对 & 允许列表         │
└─────────────────────────┬────────────────────────────────────────┘
                          │ normalized inbound event
                          ▼

│               Gateway（控制面 / 单进程常驻）                       │
│  ┌────────────────────────────────────────────────────────┐      │
│  │  Router：peer/guild/team/accountId 多级匹配             │      │

│  │  Presence / Health / Heartbeat                         │      │

                          │ routed: agentId + session + prompt
                          ▼

│   createAgentSession → session.prompt(text, {images})            │
│   ┌────────────────────────────────────────────────────┐         │
│   │ 事件订阅：message_* / tool_execution_* / turn_*    │         │
│   │ 扩展（Hooks）：compaction-safeguard / ctx-pruning │         │
│   │ 故障转移：多 authProfile 冷却 & 轮换              │         │
│   └────────────────────────────────────────────────────┘         │
└──────────────────────┬───────────────────────────────────────────┘
                       │ 工具调用
                       ▼
┌──────────────────────────────────────────────────────────────────┐
│                 Tools / Skills / Plugins                          │
│   内置工具：exec / read / write / browser / message / cron ...   │
│   渠道工具：按渠道注入（e.g. telegram.reply_with_markdown）       │
│   插件工具：ClawHub 安装的第三方（e.g. Lobster / OpenProse）     │

└──────────────────────┬───────────────────────────────────────────┘
                       │ 结果回流
                       ▼
┌──────────────────────────────────────────────────────────────────┐
│                   Sandbox & Filesystem                            │
│   Docker（默认）/ SSH / OpenShell   +   workspace-<agentId>/     │
└──────────────────────────────────────────────────────────────────┘
```

## 1.4 关键目录速查（推断 + 常见惯例）

> ⚠️ 仓库完整目录树需要 clone 后实际看，这里给出**推断 + 惯例**的对照，帮助你第一次打开仓库时不迷路。

```
openclaw/
├── package.json                # pnpm workspace 根
├── pnpm-workspace.yaml
├── VISION.md / README.md
├── ui/                         # Control UI 前端（React/Vite，构建到 dist/control-ui）
├── dist/                       # 构建产物
│
├── packages/                   # 各个子包（典型 monorepo 结构）
│   ├── cli/                    # openclaw 命令行入口（onboard/gateway/agents/doctor）
│   ├── gateway/                # 控制面：HTTP + WS + 路由 + 协议
│   ├── agent/                  # Pi 内嵌封装：runEmbeddedPiAgent
│   ├── channels/               # 各渠道适配器
│   │   ├── telegram/
│   │   ├── whatsapp/
│   │   ├── discord/
│   │   └── ...
│   ├── tools/                  # 内置工具实现
│   ├── sessions/               # JSONL 会话存储与压缩
```

## 1.5 源码阅读"路线地图"

你拿到仓库后，**不要**从 `package.json` 开始一行行扫。推荐按以下顺序切入，每步你会在后续章节看到详细拆解：

```
Step 1: 找入口
   → packages/cli/src/bin.ts（openclaw 命令入口）
   → 定位到 openclaw gateway 子命令

Step 2: Gateway 启动
   → packages/gateway/src/server.ts
   → 看 HTTP + WS 是怎么在同一端口 18789 起起来的

Step 3: 一条消息的一生
   → Telegram Webhook 入口 →
   → Channels 归一化 →
   → Gateway Router 匹配 agentId →
   → Pi Agent 执行 →
   → Tool 调用 →
   → 回流到渠道

Step 4: 扩展点
   → packages/plugins 如何装载
   → SKILL.md 如何拼进 system prompt
```

## 1.6 本章小结

- OpenClaw 是一个**自托管 + 多渠道 + Gateway 中心化**的 AI Agent 产品
- 核心哲学四条：Gateway 作为唯一事实源、内嵌 Pi、确定性路由、三层扩展分离
- 全栈分五层：**用户入口 → Channels → Gateway → Pi Agent → Tools/Skills/Plugins**
- 源码入口：`packages/cli/src/bin.ts`，一路追到 Gateway 启动、消息路由、Agent 执行
